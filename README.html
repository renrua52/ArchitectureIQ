ArchitectureIQ 开发计划总览

ArchitectureIQ 是一个用于衡量 LLM 和人类是否理解基础模型结构、优化器、损失函数与训练预算关系的可控 benchmark。 当前版本已经形成 Python 包、CLI、三类合成数据族、候选集生成、真实训练、显著性筛题、prompt 渲染、LLM eval 和 Streamlit inspector 的端到端雏形。

分支：shaoyang 更新时间：2026-07-09 Profile：v1 Python：3.10+ 测试入口：pytest

立刻体验做题

当前导入的是一轮 65 道三选一题：bigram_lm 20 道、multivariate_regression 20 道、 univariate_regression 25 道。题目会给出合成数据集、训练预算和三个候选训练方案， 你需要判断哪个方案在对应 held-out metric 上最好。

做题界面会先隐藏真实指标和正确答案；选择后才展示排名、曲线和候选文件。Learning curves 里有 Log Y 复选框，便于查看数量级差异很大的曲线。

评测报告

本轮主结论：gpt-5.5 high 在无反馈盲答下为 81.5%； 用 tools/sequential_feedback_session.py 固定逐题反馈协议后，三次连续反馈平均为 80.5%，标准差 1.9%。

去答案题集指的是：65 道题一次性提供给模型，但去掉正确答案、真实指标、训练曲线和 summary。 这不是“一次只给一题”。只有连续反馈实验才按题号一题一题做。

盲答复现 A/B/C gpt-5.5，high effort 每个 agent 独立、一次性看到同一份 65 道去答案题 无 37/65、42/65、40/65；多数投票 39/65

原始盲答是第一批主结果；盲答复现 A/B/C 是后来用同一无反馈协议重跑，用来估计波动。 非审计连续反馈和 CLI 审计连续反馈都一题一题做，区别是后者由工具强制“先记录答案，再显示反馈”。

讨论启发

题型多样性

除了综合判断题，可以增加更窄的控制变量题，例如只改变激活函数、只改变优化器、或专门测试 memorization/random dataset。

可学习性与作弊

连续刷题会让人和模型学到 shortcut；后续评测可以区分 closed-book、few-shot、course + test 和带反馈连续作答。

答题体验

曲线重叠时需要更好的可视化控制；当前 inspector 已支持在学习曲线里切换 Log Y。

Benchmark 主流程

开发时要守住一个原则：prompt 中展示的规格和代码，必须就是后端训练和评测实际执行的东西。

1. Dataset合成数据，写入 dataset_spec.json、synthesize.py 和 tensor 文件。
2. Candidate采样模型、优化器、损失、batch size 和预算，写入候选 spec 与代码。
3. Ground Truth多 seed 真实训练，保存均值、方差、曲线和失败状态。
4. Significance筛掉差距不足、胜率不稳或结果失败的候选组合。
5. Question生成多选题，记录正确答案但不泄漏 metric。
6. Evaluation给人或 LLM 作答，用答案字母和 ground truth 对齐评分。

当前 v1 Profile

维度当前配置开发含义
数据族 univariate_regression、multivariate_regression、bigram_lm 先覆盖回归和小型语言建模，后续再扩展分类、图像或真实数据。
模型池 mlp、transformer_lm MLP 用于回归；Transformer LM 用于 bigram next-token 预测。
优化器 SGD、Adam、AdamW、RMSprop、Adagrad 足够产生学习动力学差异，也便于构造 optimizer-only 问题。
损失池 回归：MSE/L1/L2；语言：CE/L1/L2 用于测试模型是否理解 objective 与 selection metric 的区别。
训练预算 1024 到 40960 total samples seen 所有候选必须在同题内共享预算，避免用训练更久混淆模型更好。
显著性 gap_min=0.05、win_rate_min=0.7、非重叠可选 保证每题只有一个稳定最佳选项。

开发计划

近期：把 v1 做扎实

  • 继续补足基础模块单元测试，重点保护 spec、路径、候选集、问题集、prompt parity。
  • 给 CLI 增加更顺手的非交互 pin 参数，减少必须进 -i 的场景。
  • 建立小规模 smoke artifact，供 inspector 和 prompt renderer 稳定截图测试。
  • 把 README、HTML 总览、plan-v2 的状态同步起来，避免路线图漂移。

中期：扩大 benchmark 能力

  • 引入更多 question type：data_only、跨预算对比、受控干扰项。
  • 为不同 family 增加更细的显著性阈值和失败诊断。
  • 丰富 LLM eval 汇总报告：按 family/type/budget/model 分桶分析。
  • 让 inspector 成为稳定的人类答题和 artifact 审核界面。

长期：从合成走向复杂任务

  • 扩展分类、图像、序列和更真实的训练机制。
  • 引入更多模型类型，例如 CNN、RNN 或小型 modern sequence model。
  • 构建公开 benchmark release：固定题集、隐藏答案、可重复评分。
  • 比较人类、不同 LLM、不同 prompt 风格在 architecture intuition 上的差异。

持续质量线

  • 每次改生成逻辑都要有 determinism 测试。
  • 每次改 prompt 都要有渲染内容和 inspector parity 测试。
  • 每次改训练代码都要跑 smoke ground truth，而不是只测 JSON。
  • 截图要覆盖文档、question inspector、关键宽度和关键状态。

风险和下一步判断

风险表现处理方式
题目过易或过难 LLM 总是秒选或随机猜 按 family/type/budget 分桶分析，并调整候选采样和显著性阈值。
prompt 泄漏答案 模型不需要理解训练机制也能推断 保持 prompt 不含 metric、seed 统计和曲线；增加 prompt snapshot 测试。
训练开销扩大 候选池生成慢，CI 难以覆盖 区分 fast unit、smoke ground truth、offline benchmark 三层验证。
扩展时 schema 漂移 旧 artifact 或 inspector 读取失败 保持稳定 envelope，把 family/type 变化放到 nested fields。

下一步建议：先把 v1 的小规模 artifact 固化成 smoke fixture，再推进 inspector 截图回归和 LLM eval 报告分桶。
